# Quality Guidelines

> 后端代码质量规范。内部项目精简版。

---

## Overview

后端位于 `backend/`,module 为 `github.com/Wei-Shaw/sub2api`,Go 1.26 + Gin + Ent + Google Wire + Viper,数据库 PostgreSQL 专用 + Redis。

分层与依赖方向(由 golangci-lint depguard 强制,违反直接 CI fail):

```
handler → service ← repository
              ↑
   repository 接口定义在 service 包内,repository 包实现它(依赖倒置)
```

- `internal/handler/`:HTTP 层,只做参数绑定、调 service、写响应;DTO 与 mapper 在 `internal/handler/dto/`。
- `internal/service/`:业务逻辑 + 领域模型 + repository 接口定义(如 `AccountRepository`,见 `backend/internal/service/account_service.go`)。
- `internal/repository/`:Ent ORM + 原生 SQL 实现 service 接口,并把持久层错误翻译为 service 层错误。
- `internal/pkg/`:可复用基础设施(`errors`、`logger`、`response`、`pagination`、`ctxkey` 等);`internal/util/` 只放少量小工具。
- `ent/`:Ent 生成代码,schema 在 `backend/ent/schema/`;`migrations/`:嵌入式 SQL 迁移。

## Naming

- 文件名 snake_case,按领域概念聚簇:`account_service.go` / `account_repo.go` / `account_repo_test.go`。
- service 层接口命名 `XxxRepository` / `XxxCache`;repository 实现为非导出 struct(`accountRepository`),构造函数 `NewXxxService` / `NewXxxRepository`,Wire provider 命名 `ProvideXxx`(见 `backend/internal/service/wire.go`)。
- Sentinel error 用 `Err` 前缀 + `internal/pkg/errors` 构造器,Reason 为 SCREAMING_SNAKE(摘自 `backend/internal/service/account_service.go`):

```go
var ErrAccountNotFound = infraerrors.NotFound("ACCOUNT_NOT_FOUND", "account not found")
```

- 缩写词遵循 `.golangci.yml` staticcheck `initialisms` 列表(API/ID/HTTP/URL/DB 等);ST1003 已关闭,存量 `ApiKey` 风格不强制重命名,新代码优先 `APIKey`。
- context key 一律用 `internal/pkg/ctxkey` 的类型化 `Key`,禁止裸 string(staticcheck SA1029,见 `backend/internal/pkg/ctxkey/`)。

## Hard Rules(项目特有,必须遵守)

1. **service 不得 import repository / gorm / go-redis**;**handler 不得 import repository / gorm / go-redis**。由 `backend/.golangci.yml` 的 depguard 规则 `service-no-repository` / `handler-no-repository` 强制。仅少数 ops_* 文件与 `service/wire.go` 在白名单中,不要往白名单加文件。
2. **repository 必须返回 service 层错误**,不泄漏 ent 错误。统一走 `translatePersistenceError`(见 `backend/internal/repository/account_repo.go`):

```go
if err != nil {
    return translatePersistenceError(err, service.ErrAccountNotFound, nil)
}
```

3. **错误统一用 `internal/pkg/errors.ApplicationError`**(Code 为 HTTP 状态码 + Reason + Message),构造器 `NotFound` / `BadRequest` 等;包装底层错误用 `WithCause`。详见 error-handling.md。
4. **handler 响应统一走 `internal/pkg/response`**(`Success` / `ErrorFrom` / `BadRequest` / `Unauthorized`…),不要手写错误 `c.JSON`。`response.ErrorFrom(c, err)` 会解析 ApplicationError 自动映射状态码(用法见 `backend/internal/handler/announcement_handler.go`)。
5. **生成代码禁止手改**:`ent/`(文件头 `Code generated by ent, DO NOT EDIT`)与 `cmd/server/wire_gen.go`。改 schema 或依赖图后重新生成:

```bash
cd backend && make generate   # = go generate ./ent && go generate ./cmd/server
```

6. **数据库迁移**:新迁移放 `backend/migrations/NNN_描述.sql`(递增编号),通过 `embed.FS` 打进二进制,启动时在 PostgreSQL advisory lock 下按文件名顺序执行(`backend/internal/repository/migrations_runner.go`)。**已应用的迁移文件内容不可再修改**——runner 记录 SHA256 checksum 并校验。不能跑在事务里的迁移用 `_notx.sql` 后缀。
7. **依赖注入只走 Wire**:新 service/repository 要在对应包的 `wire.go` ProviderSet 注册,然后 `make generate`;不要手写全局单例。
8. **数据库只支持 PostgreSQL**,SQL 可以用 PG 特有语法(sqljson、partial index、advisory lock 等),无需兼容 MySQL/SQLite。

## Forbidden Patterns

- `interface{}` — gofmt rewrite-rule 会强制改写为 `any`(`.golangci.yml` formatters 配置)。
- 忽略错误返回值 — errcheck 开启且 `check-type-assertions: true`,类型断言必须用 `v, ok :=` 双返回值形式。
- service / handler 直接 import `internal/repository`、`gorm.io/gorm`、`github.com/redis/go-redis/v9`(depguard,见 Hard Rules 1)。Redis 访问在 service 层通过接口抽象,由 repository 层实现。
- 手改 `ent/`、`wire_gen.go` 等生成文件。
- 修改已应用的 `migrations/*.sql`(checksum 校验会让所有已部署实例启动失败)。
- `context.Value` 使用内置 string 作为 key(SA1029),必须用 `ctxkey.Key`。
- 新增未在 `.golangci.yml` 白名单内的 lint 例外;需要豁免时改配置走 review,不要满地 `//nolint`。

## Linting

配置文件:`backend/.golangci.yml`(golangci-lint **v2**,CI 用 v2.7,安装见 DEV_GUIDE.md)。

启用的 linters:`depguard`、`errcheck`、`gosec`、`govet`、`ineffassign`、`staticcheck`、`unused`;formatter 为 `gofmt`(simplify 关闭 + rewrite rules)。

```bash
cd backend && golangci-lint run ./...
```

要点:

- staticcheck `checks: all`,仅关闭 ST1000/ST1003/ST1020/ST1021/ST1022(注释与命名格式类)。
- gosec 仅报 high severity + high confidence,G101/G104/G404 等已排除——不要再手动 `#nosec` 这些。
- `unused.generated-is-used: true` 必须保持为 true(ent 生成 13 万+ 行代码)。

## Testing

测试按 build tag 分三层,**单元测试文件第一行必须写 `//go:build unit`**(无 tag 的 `go test ./...` 不会运行它们):

| 命令 | 内容 | 依赖 |
|------|------|------|
| `make test-unit` = `go test -tags=unit ./...` | 单元测试(主体,270+ 文件) | 无 |
| `make test-integration` = `go test -tags=integration ./...` | 集成测试(真实 PG + Redis) | Docker |
| `make test-e2e-local` = `go test -tags=e2e -v -timeout=300s ./internal/integration/...` | 端到端 | Docker |
| `make test` | `go test ./...` + `golangci-lint run ./...` | — |

约定:

- 命名:集成测试文件用 `*_integration_test.go` + `//go:build integration`;单元测试就近放在被测文件旁。
- 集成测试基础设施:testcontainers 启动 `postgres:18.1-alpine3.23` 与 `redis:8.4-alpine`,TestMain 见 `backend/internal/repository/integration_harness_test.go`,新集成测试复用该 harness,不要自己起容器。
- 单元测试工具链:`testify/require` 断言;Redis 用 `miniredis`,SQL 用 `go-sqlmock`(用例见 `backend/internal/service/ops_metrics_collector_test.go`)。
- 测试夹具用 `internal/testutil`(`NewTestUser` / `NewTestAccount`,functional options 覆盖默认值,见 `backend/internal/testutil/fixtures.go`);HTTP 测试辅助在 `testutil/httptest.go`(已设 `gin.TestMode`)。
- 给 service 接口新增方法后,必须补全所有实现该接口的 test stub,否则编译失败(常见坑,见 DEV_GUIDE.md)。
- Windows 本地无 make 时直接执行 Makefile 中的原始命令。

## Code Review Checklist

- [ ] 依赖方向正确:handler/service 未 import repository、gorm、go-redis
- [ ] repository 返回的是 service 层 sentinel error,未泄漏 ent 错误
- [ ] handler 响应走 `internal/pkg/response`,错误走 `ErrorFrom`
- [ ] 新增/修改接口后,Wire ProviderSet 已更新并重新 `make generate`
- [ ] 迁移是新增编号文件,未改动历史迁移
- [ ] 新测试文件带正确 build tag;`go test -tags=unit ./...` 与 `golangci-lint run ./...` 本地通过
